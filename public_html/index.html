UAT Agent Simlation Framework

Web User Agent Modeling

This prototype was implemented using a simulation based web application test automation framework, that facilitates the automated execution of interactions between simulated web user agents and a target web applications. The resulting models can then be submitted to a cloud based simulator service which would spawns virtual web user agents that can interact with a targeted web applications.

While this prototype does not set out to address all the would be challenges of automated acceptance testing, it does however, effectively illustrates that the basic interactions between a HTML website and it's users, can be standardized in a functional framework that would allowing software testers that do not have programming/scripting skills to benefit from tools designed to augment their testing with automation.

This would allow for not just programmers but other key stake holders involved in the iterative delivery of software to benefit from AGILE methodologies, by being equipped with the tools they need to actively model and execute web user test simulations independent of software developers.

Extending User Stories

The framework builds on the widely accepted AGILE concept of a User Story, which has effectively narrowed the gap between requirements gathering, development and testing by being a common artifact that is used throughout the various stages of the Software Development LIfe Cycle (SDLC).

In many frameworks, 'user stories' have widely regarded as 'executable' requirements that enables AGILE development approaches such as Behavior Driven Development (BDD).

User story elements such as As a to define a user's ROLE, So that to define a user's BELIEF, Feature

But not without the involvement of a programmer to 'code' the steps that linked to each user story. Furthermore, the

step codes are invariably written by programmers for each unique web application to be tested.

While developer's may try to 're-use' snippets and keep the code they write for testing DRY (do not rewrite), there is still much more potential to further reduce re-working and improve efficiency.

Mapping User Story Elements to BDI Modelling

At the core of agent based modeling are the conceptual elements of BDI (belief, desire, intent). User stories contain elements that can be mapped to the BDI. The framework takes advantage of this, effectively extending the conceptual use of user story elements to cover the basics of the modeling required.

Data Instantiation

Preparing and using test data for acceptance tests is always a challenge. The "Given that" and "Background" sections of a user story is meant to facilitate the 'staging' of data, but given the way that user story 'steps' are widely used today for automated testing, where data declarations are being made in the middle of a step, it invariably results in step definitions that are not 'reusable'.

To make tests reusable, the framework posits that all data declarations should be made separately from the step definitions. This endows the simulated user with the ability for example to 'know' his name before being required to enter it on a web page form.

This prototype also illustrates how data from external or even third party sources could be integrated into a more comprehensive solution that can 'tell' a modeled web user what it needs to know before step execution.

'True' User Steps

The 'step' modeling feature is where the rubber meets the road in the framework. It represents the critical "When I" prefixed lines of a user story. The intuitive interfaces makes it easy for non-programmers to define what a web user agent should do in specific scenarios. The prototype also illustrates several seemingly novel concepts that the framework introduces, including:

AGENT ACTUATORS:

TRUE BLACK BOX TESTING:

When using a web application, real users don't see the attributes and other inner workings of the HTML elements on a web page. Automated behavioral/functional tests should therefore strictly emulate only the capabilities of real users to be a meaningful 'user' test

SPATIAL BROWSER AWARENESS:

When real users are being guided on how to navigate and find specific elements on a web page, they take instruction based on first finding distinct location or unique identifiable visual cues on the web page, then find other elements that they need to interact with using spatially descriptive language such as 'BESIDE', 'BELOW', 'TO THE LEFT' etc.
These spatially useful descriptive terms referred to in the framework as Relative Positions are a key way to provide instructions in step definitions on how a simulated web user can find a specific Target on which to perform an Action

FOCUS ON USER EXPERIENCE:

Web pages can change while the described user experience remains intact. Automated user tests should be able to validate the basic elements of described user experience.

User Based Assertions

To truly simulate user acceptance testing, the framework focuses on facilitating assertions that a real user would be able to determine from just interacting with the resulting web pages after defined steps are executed.

AGENT SENSORS:

the framework posits that web users have specific 'sensor' capabilities that make them able to detect what transpires on a web page subsequent to the execution of steps, these include the ability to 'SEE', 'HEAR', and 'FEEL' stimuli produces by a web page.

The full implementation of all the sensors that the framework proposes goes beyond the scope of this research, but the named capabilities are included with the hope that the open source community will not only contribute to the continued development of the framework to its full potential, but also to fork and customize the framework to meet other needs.

This prototype implements the 'SEE' interface of the framework which is currently limited to web page element types and text, but there is great potential to extend this to include seeing properties such as defined 'COLOR' and 'SIZE' descriptions to name a few.
